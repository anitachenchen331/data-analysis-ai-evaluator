# 产品公开信息调研

## 目的

根据用户提供的产品名称、链接或材料，主动调研产品声称支持的数据分析能力，再与用户确认本轮评测目标。

不要仅依赖用户的初始描述来定义评测能力。用户通常只知道“想测一下”，而不是完整知道产品能力边界。

## 信息来源优先级

| 优先级 | 来源 | 默认可信度 | 用途 |
|---|---|---:|---|
| P0 | 用户提供的官方材料、后台截图、私有说明 | 高 | 确认实际可用能力和环境 |
| P1 | 官方网站、官方文档、官方 API 或 CLI 文档、官方发布说明 | 高 | 识别产品正式能力主张 |
| P2 | 官方代码仓库、官方示例、官方模板、官方演示项目 | 高-中 | 验证能力是否有实际接口或示例 |
| P3 | 官方博客、教程、白皮书、发布活动材料 | 中 | 理解定位和使用场景 |
| P4 | 第三方评测、媒体文章、社区教程 | 中-低 | 辅助理解，不作为唯一依据 |
| P5 | 论坛、issue、社交媒体讨论 | 低 | 识别风险和限制 |

## 调研输出

输出 `product_eval_brief.md`，包括：

- 产品名称、版本或调研日期。
- 产品主张能力。
- 每项能力的证据来源。
- 每项能力的可信度。
- 可评测能力。
- 暂不可评测能力。
- 需要用户补充的信息。
- 建议实验目的。

## 能力分类

用通用能力标签归类，不绑定具体行业：

- 数据连接
- 数据问答
- 查询生成
- 查询执行
- 语义层或业务上下文
- 指标解释
- 图表生成
- 看板生成
- 异动诊断
- 归因分析
- 自动报告
- 多轮分析
- 记忆或样例复用
- 权限治理
- 错误修复
- 可追溯性

## 用户确认问题

调研后必须问用户确认：

- 这些能力是否符合你对产品的理解？
- 本轮要重点评测哪些能力？
- 哪些能力明确不测？
- 本轮目的是选型、验收、对比、复测、风险审计还是产品改进？
- 是否有私有文档、账号或演示环境可以提高可信度？

## 输出语气

用证据驱动表述：

```text
我从官方文档看到该产品声称支持“图表生成”，可信度高；从社区文章看到有人使用它做“异常解释”，可信度中低，需要你确认是否纳入评测。
```
