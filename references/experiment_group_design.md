# 实验组设计

## 原则

不要固定套用某一种实验组结构。根据用户确认的实验目的，采用“最小化原则”和“最优原则”设计实验组。

- 最小化原则：只设计能回答当前实验问题的必要组。
- 最优原则：保留能解释差异来源的对照组。

## 常见目标与推荐组

| 实验目标 | 推荐实验组 |
|---|---|
| 判断产品是否比普通模型好 | 普通模型组 vs 产品组 |
| 判断业务上下文是否有价值 | 无上下文组 vs 上下文组 |
| 判断完整工具链是否有额外价值 | 上下文组 vs 完整产品链路组 |
| 比较两个产品 | 产品 A 组 vs 产品 B 组，必要时加普通模型组 |
| 验证新版本是否改善 | 旧版本组 vs 新版本组 |
| 评估权限治理 | 正常请求组 vs 受限权限组 vs 攻击请求组 |
| 评估稳定性 | 同一产品多次重复运行 |
| 评估多轮能力 | 单轮组 vs 多轮组 |
| 评估图表能力 | 查询结果组 vs 图表产物组 |
| 评估诊断能力 | 只给结果组 vs 给可用证据组 vs 产品诊断组 |

## 每组必须定义

- 组名。
- 组目的。
- 允许读取的信息。
- 禁止读取的信息。
- 允许使用的工具。
- 禁止使用的工具。
- 输出文件路径。
- 是否允许联网。
- 是否允许看历史结果。
- 是否需要记录工具链证据。

## 污染隔离

实验运行组禁止读取：

- 内部评分 key。
- 标准答案。
- 标准查询或校验脚本。
- 其他实验组结果。
- 评分报告。

评分组才允许读取内部材料。

## 输出建议

生成 `experiment_and_scoring_plan.md` 和 `group_prompts.md`。

每个 prompt 都要写清楚：

- 当前组身份。
- 允许文件。
- 禁止文件。
- 允许工具。
- 禁止工具。
- 输出格式。
- 结果路径。
- 需要记录的证据。
